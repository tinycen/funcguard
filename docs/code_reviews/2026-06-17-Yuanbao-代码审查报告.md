### 问题 1：JSON 解析失败会中断整列数据转换

**文件**：`funcguard/pd_utils/convert_utils.py`
**行号**：`load_json` 函数（第 155-163 行）
**级别**：🔴 严重bug

**问题描述**：
`load_json` 函数使用 `apply(lambda x: json_loads(x, empty_to_dict))` 对整列执行 JSON 解析。`json_loads` 在解析失败时直接 `raise ValueError`，导致单行数据异常时整列转换崩溃。若 DataFrame 中某行 JSON 格式错误，整个列处理流程将中断，无法定位具体问题行。

**代码片段**：
```python
def load_json(
    df: pd.DataFrame,
    columns: List[str],
    empty_to_dict: bool = True,
) -> pd.DataFrame:
    for column in columns:
        df[column] = df[column].apply(lambda x: json_loads(x, empty_to_dict))
    return df
```

**修复建议**：
将 `json_loads` 的错误处理方式改为返回原始值或填充默认值，而非抛出异常。同时在 `load_json` 中增加 try-except 兜底：

```python
def load_json(
    df: pd.DataFrame,
    columns: List[str],
    empty_to_dict: bool = True,
) -> pd.DataFrame:
    for column in columns:
        def safe_loads(x):
            try:
                return json_loads(x, empty_to_dict)
            except ValueError:
                # 解析失败时返回原始值，避免整列中断
                return x
        df[column] = df[column].apply(safe_loads)
    return df
```

### 问题 1：`log_utils.py` 模块多次加载会触发 `ValueError`

**文件**：`funcguard/log_utils.py`
**行号**：第 35-37 行
**级别**：🟠 注意问题

**问题描述**：
`SUCCESS_LEVEL = 25` 和 `PROGRESS_LEVEL = 35` 在模块顶层定义，每次 import 时都会调用 `logging.addLevelName()`。Python 不会对重复添加的 level name 报错（底层是 dict 操作），但这会导致模块状态不可预测。更严重的是，如果模块被 reload()，会触发 `ValueError: level name SUCCESS already added`。此外 `SuccessLogger` 类定义后又被 monkey patch 覆盖（第 78-90 行），逻辑冗余。

**代码片段**：
```python
SUCCESS_LEVEL = 25
PROGRESS_LEVEL = 35
logging.addLevelName(SUCCESS_LEVEL, "SUCCESS")
logging.addLevelName(PROGRESS_LEVEL, "PROGRESS")


class SuccessLogger(logging.Logger):
    def success(self, message, *args, **kwargs) -> None: ...
    def progress(self, message, *args, **kwargs) -> None: ...

# 后面又 monkey patch
if not hasattr(logging.Logger, "success"):
    setattr(logging.Logger, "success", _logger_success)
```

**修复建议**：
使用 `try-except` 保护 `addLevelName` 调用，或在函数内用 `setattr` 方式安全注入自定义 level，避免模块重复加载风险。同时清理冗余的类定义，monkey patch 应作为唯一扩展机制。

### 问题 4：`ask_select` 超时返回后 `input_thread` 仍在后台运行

**文件**：`funcguard/core.py`
**行号**：`ask_select` 函数约第 159 行
**级别**：🟠 注意问题

**问题描述**：
超时返回时调用 `thread.join(0)`，线程在 `daemon=True` 下会随进程终止被强制销毁。但若在非 daemon 场景使用（或线程内 `input()` 被阻塞在系统层），可能导致资源泄漏。此外，超时后 `result_queue.get_nowait()` 抛 `Empty` 异常后直接返回，`input_thread` 中的 `sys.stdin` 可能残留未消费的行，影响下一次 `input()` 调用。

**修复建议**：
使用 `thread.join(timeout)` 等待线程真正退出，或在超时分支主动关闭 stdin：

```python
thread.join(timeout=1)  # 等待最多1秒
```

### 问题 3：`core.py` `retry_function` 参数顺序导致使用上的歧义

**文件**：`funcguard/core.py`
**行号**：第 47 行
**级别**：🟡 一般问题

**问题描述**：
`def retry_function(func, max_retries=5, execute_timeout=90, task_name="", *args, **kwargs)` 中，`task_name` 位于可变参数之前，在实际调用 `retry_function(func, max_retries, execute_timeout, task_name, *args, **kwargs)` 时，如果调用者想传递额外参数但又不想指定 `task_name`，会产生歧义。Python 参数解析规则要求带默认值的参数必须在 `*args` 之后，这是语义上的反面模式（antipattern）。

**修复建议**：
将 `task_name` 移入 `kwargs` 中，或使用 `TypedDict` 定义配置结构：

```python
def retry_function(
    func,
    max_retries: int = 5,
    execute_timeout: int = 90,
    *args,
    task_name: str = "",
    **kwargs,
):
```
