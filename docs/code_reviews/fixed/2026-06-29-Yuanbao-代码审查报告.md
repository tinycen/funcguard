## 🔴 严重bug (CRITICAL)

### CR-1: `json_loads()` 文档说明与实现不符，级联崩溃风险

**文件**: `funcguard/pd_utils/json_utils/json_parser.py` 第 25–26 行

**问题描述**:

文档字符串明确写明：
> 返回：解析后的Python对象，**或原始值（如果解析失败）**

但实际代码在 JSON 解析失败时抛出异常：

```python
try:
    return json.loads(value)
except (json.JSONDecodeError, TypeError):
    # 解析失败，返回原值   ← 注释说的是返回原值
    raise ValueError(f"JSON解析错误：{value}")  ← 实际抛出了异常
```

**危害**: `json_loads` 被 `load_json()` 中的 `.apply()` 调用，应用于整个 DataFrame 列。一旦某行含非法 JSON 字符串，整个 DataFrame 处理流程会直接崩溃，而不是跳过该行继续处理。

**修复建议**:

```python
except (json.JSONDecodeError, TypeError):
    # 解析失败，返回原值
    return value  # 而非 raise ValueError(...)
```

### CR-2: `send_request()` 日志记录未脱敏敏感认证信息

**文件**: `funcguard/tools.py` 第 159–172 行

**问题描述**:

当 `request_log.save_path` 设置时，代码将完整请求头写入日志文件：

```python
save_fields = [
    ("save_method", "method", method),
    ("save_url", "url", url),
    ("save_headers", "headers", headers),   # ← 包含 Authorization / Cookie 等敏感头
    ("save_body", "body", payload),
    ("save_response", "response", result),
]
for attr, key, value in save_fields:
    if getattr(request_log, attr):
        res_log[key] = value
with open(request_log.save_path, "w", encoding="utf-8") as f:
    json.dump(res_log, f, ensure_ascii=False, indent=4)
```

若请求头中包含 `Authorization: Bearer xxx`、`Cookie`、`X-API-Key` 等认证信息，会以明文形式写入磁盘日志，带来信息泄露风险。

**修复建议**:

在写入日志前过滤敏感头字段：

```python
SENSITIVE_HEADERS = {"authorization", "cookie", "x-api-key", "x-auth-token"}
safe_headers = {
    k: ("***REDACTED***" if k.lower() in SENSITIVE_HEADERS else v)
    for k, v in headers.items()
}
res_log["headers"] = safe_headers
```

## 🟠 注意问题 (WARNING)

### WR-1: `is_valid_ip()` 仅支持 IPv4，IPv6 地址会静默误判

**文件**: `funcguard/ip_utils.py` 第 47–62 行

**问题描述**:

`is_valid_ip()` 使用 `split('.')` 和 `isdigit()` 验证 IP，只能正确处理 IPv4。IPv6 地址（如 `2001:0db8::1`）会直接返回 `False`，导致 `get_public_ip()` 对 IPv6 环境静默返回 `None`。

**修复建议**:

使用 `ipaddress` 标准库：

```python
import ipaddress

def is_valid_ip(ip_string: str) -> bool:
    try:
        ipaddress.ip_address(ip_string)
        return True
    except ValueError:
        return False
```

### WR-2: `retry_function()` 无差别捕获所有异常，无法区分重试场景

**文件**: `funcguard/core.py` 第 41–68 行

**问题描述**:

代码对所有异常都执行重试逻辑：

```python
except Exception as e:
    last_exception = e
    retry_count += 1
    print(e)
    if "TimeoutError" in str(e):   # ← 只能识别包含"TimeoutError"的异常
        # ... 超时扩展逻辑
    print(f"{task_name} : {func.__name__} 请求失败，正在重试...")
```

问题：
- 非超时异常（如 400 Bad Request、401 Unauthorized、参数错误）也会被重试，浪费资源。
- 仅通过字符串匹配 `"TimeoutError" in str(e)` 判断类型不可靠。

**修复建议**:

```python
except FuncguardTimeoutError:
    # 超时异常 → 扩展超时重试
    pass
except (requests.RequestException, ConnectionError):
    # 网络异常 → 重试
    pass
except Exception:
    # 其他异常 → 立即失败，不重试
    raise
```

### WR-3: `fill_na()` 原地修改 DataFrame，行为与文档描述不符

**文件**: `funcguard/pd_utils/fill_utils.py` 第 33–34 行

**问题描述**:

文档字符串注明：
> 注意：本函数直接修改传入的 DataFrame，不返回副本。如需保留原数据，请在调用前使用 df.copy()。

但函数签名末尾有 `return df`，且 `pd_fill_na` 在 `__init__.py` 中被作为主要导出接口。用户容易误以为返回了新 DataFrame 而不调用 `.copy()`，导致意外修改原始数据。

**修复建议**:

考虑改为返回副本（破坏性变更需注意兼容性），或在所有填充分支统一加上显式注释强调此行为。

### WR-4: `get_public_ip()` 异常处理过于宽泛，掩盖潜在问题

**文件**: `funcguard/ip_utils.py` 第 35–45 行

**问题描述**:

```python
try:
    response = requests.get(service, timeout=5)
    if response.status_code == 200:
        ip = response.text.strip()
        if is_valid_ip(ip):
            return ip
except Exception:   # ← 所有异常统一忽略
    continue
return None
```

若后续 IP 服务返回非 200 状态码但有内容（如 403 返回错误 JSON），会被静默跳过，用户无法区分是网络问题还是服务拒绝。

**修复建议**:

在所有服务均失败后，打印一条警告信息便于排查：

```python
print("警告: 所有公网IP查询服务均失败，请检查网络连接")
return None
```

### WR-5: 类型注解使用旧式 `Union[...]` / `Optional[...]`，与项目要求不一致

**文件**: 多处（`core.py`, `tools.py`, `time_utils.py`, `pd_utils/` 目录下几乎所有文件）

**问题描述**:

Python 3.10+ 支持 PEP 604 的联合类型语法 `X | Y`，项目 `setup.py` 中 `python_requires='>=3.10'`，但大量文件仍使用旧式写法：

```python
# 旧式（多处存在）
from typing import Union, Optional
def foo(x: Optional[int], y: Union[str, list]) -> Union[Dict, str]:

# 现代写法（推荐）
def foo(x: int | None, y: str | list) -> dict | str:
```

**修复建议**:

统一使用现代联合类型语法，移除 `from typing import Union, Optional`，减少导入噪音。

## 🟡 一般问题 (INFO)

### IN-1: `cal_date_diff()` 参数 `nat` 类型为 `int | None`，但函数返回值可能不匹配

**文件**: `funcguard/pd_utils/date_utils.py` 第 61 行

**问题描述**:

`nat` 参数接收 `int` 类型的填充值，但函数最终 `return result.astype(float if decimal_places > 0 else int)` 会统一转为数值类型。若 `nat=0`，而某行日期为 NaT，填充结果为 `0.0`，与合法的 `0`（代表0小时差）无法区分。

**修复建议**:

考虑使用 `np.nan` 作为 NaT 填充值，或在返回值中增加元组形式同时返回掩码信息。

### IN-2: `convert_decimal()` 对非 object 类型列静默跳过，无提示

**文件**: `funcguard/pd_utils/convert_utils.py` 第 213–215 行

**问题描述**:

```python
if column not in df.columns or df[column].dtype != object:
    continue  # ← 静默跳过，用户不知道哪些列被跳过了
```

### IN-3: 字符串操作使用 `.apply(lambda)` 而非向量化方法

**文件**: `funcguard/pd_utils/statistics/mask_utils.py` 第 35–50 行（`_is_empty` / `_is_not_empty`）；`funcguard/pd_utils/convert_utils.py` 第 264–267 行（`parse_datetime`）

**问题描述**:

在 pandas 中，`.apply()` 按行迭代 Python 函数，性能远低于向量化操作。在大数据 DataFrame 下会有明显性能瓶颈。

```python
# mask_utils.py
df[column].apply(_is_empty)  # ← 逐行迭代

# convert_utils.py
df[column].apply(parse_datetime)  # ← 逐行迭代
```

**修复建议**:

对于 `_is_empty`/`_is_not_empty`，可使用 pandas 向量化逻辑替代。对于 `parse_datetime`，`pd.to_datetime()` 本身已是向量化，直接用即可（已在 `convert_str_datetime` 中正确使用），但在 `convert_str_datetime` 中的 `apply` 是备选方案，可加文档说明其使用场景。

### IN-4: `DataFrameStatistics.combine_masks` 作为实例属性赋值方式不标准

**文件**: `funcguard/pd_utils/statistics/df_statistics.py` 第 27 行

**问题描述**:

```python
class DataFrameStatistics:
    def __init__(self, df):
        ...
        self.combine_masks = combine_masks  # ← 实例属性而非类方法
```

将模块级函数直接赋值给实例属性，IDE 无法识别其类型，且语义不清晰（用户可能认为 `self.combine_masks` 是实例方法）。

**修复建议**:

```python
class DataFrameStatistics:
    def __init__(self, df):
        ...
        self._combine_masks = combine_masks  # 私有属性
    
    def combine_masks(self, masks, logic="and"):
        return self._combine_masks(masks, logic)
```

### MN-3: Pyright 类型检查存在 21 处报错（均为 pandas 3.x 类型存根严格性导致）

**文件**: `funcguard/pd_utils/` 目录

**问题描述**:

Pyright 报告 21 个类型错误，主要集中在以下模式：

- `df[column].astype(TYPE_MAPPING[target_type])` — pandas 3.x 类型存根将 `astype()` 返回值标记为 `Series | DataFrame | Unknown` 而非 `Series`
- `pd.to_datetime(...)` 返回值类型不精确
- `result_series.agg()` 返回 `NDFrame*` 而非 `Series`

这些问题属于**类型注解层面的误报**，不会影响运行时行为（pandas 本身运行正确），但长期存在会降低类型检查的可用性。

**修复建议**:

在相关函数中添加类型注解抑制注释 `# type: ignore[reportArgumentType]`，或使用 `cast()` 显式转换。
